Defining Jank

On a webpage, when

  • Scrolling isn't smooth
  • Animations stutter and hitch
  • Things slow down at random

...we call that jank.

Writing Animation Loops

The Old Way

To draw repeatedly on the screen, many devs are used to something like:

        <script type='text/javascript'>
          var left = 0;
          setInterval(function animate(){
            el = document.querySelector("#myEl");
            el.style.left = (left++) + "px";
          }, 16.66);
        </script>

Animation Timing

Avoiding Missed or Wasted Frames

requestAnimationFrame

The New Way

To align with the monitor's refresh every time, use requestAnimationFrame:

        <script type='text/javascript'>
          var left = 0;
          function animate(time) {
            el = document.querySelector("#myEl");
            el.style.left = (left++) + "px";
            requestAnimationFrame(animate);
            }
          requestAnimationFrame(animate);
        </script>
Press 'h' to highlight important sections of code (wrapped in <b>).

Why requestAnimationFrame?

  • Even if you only drop 1 frame in 16, you'll notice
  • Device framerates vary, hard-coding timers won't work
  • The browser can be smart
    • Pause non-visible animations, save battery life!
    • Throttle animations if the system's strained

With rAF you get what we call "vsync" -- perfect monitor refresh alignment

Staying in Budget

Staying Fast

You only get the time between refreshes to update the screen. We call this a frame budget.

On a 60Hz display, that's only ~16ms!

A great way to visualize this budget is the Dev Tools Timeline. Let's look at an example.

Juggling Input

Minimal Event Handlers

So you're using rAF and keeping your rAF callback short. Great!

What else can get in the way? Lots:

  • The garbage collector running
  • JS timers firing
  • XHR handling
  • Input event handling

Use Timeline to find these!

CSS Animation

Smooth Animation

Better than lightweight-JS animation? Zero-JS animation.

Most of what happens in a tab happens on one thread.

We're teaching the browser to run things on its own, even when JS is running.

One of the first is CSS animation.

This is on Chrome for Android first, everywhere else soon. Let's look at a mobile example.

Smooth Animation

When you can, use CSS animation to let the browser do the right thing:

  • CSS transitions
  • CSS keyframe animations
  • etc.

Like with rAF, the benefits are:

  • Better-timed, smoother-looking animations
  • Better battery life
  • More graceful degredation

Paint Times

The Browser Doth Churn

Slow Scrolls

Even if you're not animating anything, your page can scroll slowly.

Timeline in frame view can show you what's happening during a scroll. Odds are, you'll see long paint events.

What's going on? A brief aside on browser rendering...

Painting

The DOM is our model of the page. Every time it changes, we repaint.

Often when the viewport changes (e.g. you scroll) we need to repaint, too.

Example time.

Painting Takeaways

When you see a long paint time, first find what painted:

  • Highlight the paint event in Timeline
  • Match paint sizes to regions of the page
  • Try the Dev Tools 'Show paint rectangles' tool
  • Worst case, bisect: remove elements until the problem goes away

Then find why its slow to paint:

  • Unnecessarily painting large areas
  • Bisect heavy styles
